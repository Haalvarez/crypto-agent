¿Qué pasa en cada ciclo de 4h?

1
t = 0s
Fetch de mercado
data.py llama a Binance Public API y descarga las últimas 100 velas de 4h para cada par. Calcula RSI(14), EMA20, EMA50, ratio de volumen y cambio de la última vela. No requiere API key.
2
t = ~1s
Clasificación de régimen
regime.py carga el modelo HMM pre-entrenado desde disco (cache en memoria). Calcula 5 features estadísticas de las velas 4h: log-return, volatilidad rolling, ratio de volumen, RSI centrado y pendiente de EMA50. El HMM predice en cuál de los 4 estados está el mercado actualmente.
3
t = ~2s
Construcción del prompt
brain.py combina el bloque de régimen (cuántas horas lleva el estado actual, probabilidad de persistencia, volatilidad del estado) con los indicadores de mercado. Este texto completo es lo que recibe Claude como input.
4
t = ~3–6s
Análisis Claude (API)
Claude recibe el contexto y sigue el system prompt: evalúa el régimen primero, luego los indicadores técnicos, luego el Fear & Greed Index. Emite un bloque estructurado por activo con dirección, convicción 1-10, entrada, stop-loss, take-profit y tesis.
5
t = ~6s
Parseo de señales
El texto de Claude es parseado con regex. Se extraen símbolo, dirección, convicción, precios de entrada/stop/target. Una señal es "accionable" si convicción ≥ 7 y la dirección no es NEUTRAL.
6
t = ~7s
Ejecución (si hay señal)
executor.py verifica límites de riesgo: máximo de posiciones abiertas, pérdida diaria máxima, tamaño por operación. Si todo pasa, coloca una orden de mercado en Binance Testnet via ccxt. Guarda el trade en SQLite.
7
t = ~8s
Alertas Telegram
Se envían las señales al celular con régimen actual, convicción, precio de entrada, stop y target. Al final del ciclo llega un resumen silencioso con el estado del portafolio. El agente duerme 4 horas y repite.

componentes del sistema

🔢
Datos de mercado
data.py
▼
Llama a la API pública de Binance (sin autenticación) y construye el contexto técnico que recibirá Claude. Migrado a velas de 4h para alinearse con el timeframe del HMM.
  • 1GET /api/v3/ticker/24hr → precio actual y cambio 24h
  • 2GET /api/v3/klines?interval=4h&limit=100 → velas OHLCV
  • 3Calcula RSI(14) con Wilder smoothing sobre los closes
  • 4EMA20, EMA50 → determina tendencia (ALCISTA/BAJISTA)
  • 5vol_ratio = volumen actual / MA20 del volumen
requests pandas Binance Public API
🎯
Clasificador de Régimen
regime.py · models/hmm_*.pkl
▼
Carga un modelo GaussianHMM pre-entrenado sobre 2 años de datos 4h. Detecta en cuál de los 4 estados estadísticos está el mercado ahora mismo. El modelo se cachea en memoria — no hay I/O de disco en cada ciclo.
  • 1Carga hmm_{SYMBOL}.pkl con joblib (primera vez)
  • 2Computa 5 features: log_return, vol_rolling_20, vol_ratio, RSI centrado, EMA50 slope
  • 3StandardScaler normaliza las features (mismo scaler del entrenamiento)
  • 4model.predict() → secuencia de estados por Viterbi
  • 5Cuenta barras consecutivas en el estado actual, extrae probabilidad de persistencia de la transmat

⚡
Ejecutor de Órdenes
executor.py · trades.db
▼
Toma las señales accionables de Claude y las convierte en órdenes reales en Binance Testnet. Antes de ejecutar, verifica todas las reglas de riesgo hardcodeadas en config.py. Cada trade queda registrado en SQLite.
  • 1¿señal.accionable AND convicción ≥ 7? → continuar
  • 2¿posiciones abiertas < MAX_OPEN_POSITIONS (2)? → continuar
  • 3¿pérdida_dia < MAX_DAILY_LOSS_USD ($15)? → continuar
  • 4Calcular quantity = MAX_TRADE_USD ($10) / precio_actual
  • 5exchange.create_order(market, buy/sell) via ccxt
  • 6INSERT INTO trades (symbol, entry, stop, target, ...)
ccxt sqlite3 Binance Testnet
📊
Sistema de Backtest
backtest/ downloader · simulator · regime_trainer
▼
Infraestructura offline para descargar datos históricos, entrenar el HMM y validar estrategias sobre datos pasados antes de exponer capital real.
  • 1downloader.py — descarga 2 años de velas 1h y 4h de Binance (730 días × 4 pares)
  • 2regime_trainer.py — entrena GaussianHMM(n=4) con 10 reinicios aleatorios sobre los datos 4h, etiqueta estados post-hoc por estadísticas, guarda modelo en models/
  • 3simulator.py — simula la estrategia EMA/RSI sobre datos históricos, reporta Sharpe, win rate, drawdown
  • 4optimizer.py — grid search sobre parámetros con validación walk-forward 80/20
  • 5report.py — genera HTML con resultados visuales
hmmlearn pandas numpy walk-forward
📱
Alertas Telegram
telegram_alerts.py
▼
Wrapper sobre la API HTTP de Telegram Bot. Envía notificaciones formateadas en HTML a un chat específico para cada evento del agente.
  • 1Startup — al arrancar, confirma pares monitoreados e intervalo
  • 2Señal — por cada par: dirección, convicción, entrada, stop, target, tesis
  • 3Ejecución — confirma la orden colocada con order ID y cantidad
  • 4Resumen 4h — régimen actual de cada par + señales + balance USDT (silent)
  • 5Error — cualquier excepción no capturada llega al celular
