CryptoAgent — Mapa del Sistema

Flujo completo · Parámetros · Escenarios · Salidas posibles

1 · Parámetros configurables
SYMBOLS
ETH · SOL · BTC
Pares monitoreados. XRP fue removido por Sharpe negativo en backtest (−0.53). Cada par es independiente.
MONITOR_INTERVAL_MINUTES
15 min
Frecuencia del ciclo de monitoreo. Cada 15 min se chequean precios, stops y régimen. No llama a Claude.
ANALYSIS_INTERVAL_MINUTES
60 min (testnet) · 240 min (prod)
Cada cuánto Claude analiza un par libre. Se resetea si el par cierra una posición.
MAX_TRADE_USD
$10
Máximo en USD por operación. La cantidad en cripto = 10 / precio_actual.
STOP_LOSS_PCT
4%
Stop fijo desde la entrada. Pérdida máxima por trade: $0.40 (4% de $10).
TAKE_PROFIT (Claude)
~8% (ratio 1:2)
Claude sugiere el target. Ratio riesgo/recompensa 1:2 → rentable con win rate > 33%.
MAX_DAILY_LOSS_USD
$15
Si las pérdidas del día acumulan $15, el agente se pausa hasta medianoche.
MIN_SIGNAL_CONVICTION
7 / 10
Claude puntúa 1–10. Solo se ejecutan señales con puntaje ≥ 7. Filtra entradas débiles.
BINANCE_TESTNET
true → false (producción)
En true usa el sandbox de Binance. Las órdenes son simuladas, sin dinero real.

3 · Arquitectura de módulos
main.py
Orquestador central. Loop principal, ciclos de monitor y análisis, servidor HTTP.
data.py
Fetch de precios 4h desde Binance. Calcula RSI, EMA20/50, tendencia, Fear&Greed.
regime.py
Carga modelo HMM (.pkl), clasifica el régimen actual de cada par. Retorna régimen, horas, persistencia.
brain.py
Llama a Claude API. Recibe contexto de mercado + régimen. Retorna señal, dirección, convicción, tesis.
executor.py
Órdenes en Binance vía ccxt. DB SQLite de trades. Check de stop/target cada ciclo.
telegram_alerts.py
Notificaciones: apertura, cierre, errores, resumen periódico.
models/
Archivos .pkl del HMM entrenados por par (BTCUSDT.pkl, ETHUSDT.pkl, SOLUSDT.pkl).
trades.db
SQLite local. Tabla trades: entrada, salida, PnL, estado. Persiste entre reinicios.
4 · Flujo principal — ciclo completo
0
Al arrancar
Inicialización
Carga variables de entorno → init_db() crea/verifica trades.db → arranca thread del servidor HTTP en puerto configurado → envía mensaje de inicio por Telegram.
1
Cada 15 min
Fetch de mercado
Binance API Descarga las últimas 100 velas de 4h para cada par. Calcula: precio actual, RSI(14), EMA20, EMA50, tendencia (EMA20 > EMA50), volumen relativo, cambio 4h.
También obtiene el Fear & Greed Index del día desde alternative.me.

Si falla: loguea error, notifica Telegram, aborta el ciclo. Reintenta en 15 min.
2
Cada 15 min
Monitor de posiciones abiertas (SL / TP)
SQLite Para cada trade con status=OPEN compara precio actual vs stop_loss y take_profit.
LONG y precio ≤ stop_loss →
LOSS — cierra en DB, registra PnL negativo, suma a daily_loss_usd, notifica Telegram, resetea timer de análisis del par.
LONG y precio ≥ take_profit →
WIN — cierra en DB, registra PnL positivo, notifica Telegram, resetea timer.
SHORT y precio ≥ stop_loss →
LOSS — igual que arriba.
SHORT y precio ≤ take_profit →
WIN — igual que arriba.
Precio entre stop y target →
Sin acción — continúa monitoreando.
Si daily_loss_usd ≥ $15 → estado halted=true. El agente no abre nuevas posiciones hasta medianoche.
3
Cada 15 min
Clasificación de régimen HMM
modelo .pkl Para cada par: descarga 200 velas 4h de la API pública de Binance (sin auth) → calcula 5 features → aplica modelo GaussianHMM pre-entrenado → algoritmo de Viterbi predice el estado actual → etiqueta el estado como régimen.

Retorna por par: régimen, horas consecutivas, probabilidad de persistencia, régimen anterior.
Si el modelo .pkl no existe para un par, el régimen queda como UNKNOWN y se omite del contexto de Claude. No bloquea el ciclo.
4
Cada 15 min
Salidas por cambio de régimen (sin Claude)
Revisa cada posición abierta contra el régimen actual. Si la tesis de entrada ya no aplica, cierra al mercado inmediatamente, sin consumir tokens.
Posición LONG + régimen ahora es BEAR_TREND →
Cierre de mercado inmediato — orden sell al precio actual.
Posición SHORT + régimen ahora es BULL_TREND →
Cierre de mercado inmediato — orden buy al precio actual.
LONG + SIDEWAYS/REVERSAL, SHORT + SIDEWAYS/REVERSAL →
Sin acción — esos regímenes son ambiguos, no se fuerza salida.
5
Cada 1h/4h por par
Análisis Claude (solo pares libres y con timer vencido)
Claude API Condiciones para llamar a Claude:
① El par no tiene posición abierta
② Pasaron ANALYSIS_INTERVAL_MINUTES desde el último análisis
③ El agente no está en estado halted

Claude recibe: contexto de mercado (precio, RSI, EMA, tendencia, volumen) + contexto de régimen HMM (régimen, horas, persistencia, régimen anterior) + Fear & Greed + instrucciones del system prompt.

Claude devuelve por par: dirección (LONG/SHORT/NEUTRAL), convicción (1–10), stop sugerido, target sugerido, tesis.
Par con posición abierta → Skip Claude — 0 tokens consumidos.
Análisis reciente (timer no vencido) → Skip Claude — muestra tiempo restante en log.
Todos los pares ocupados → Skip ciclo completo de análisis — 0 tokens.
6
Cuando Claude responde
Ejecución de señales
Para cada señal recibida de Claude:
Dirección = NEUTRAL → No hacer nada — contador "descartados" +1.
Convicción < 7 → No ejecutar — señal débil.
Ya hay posición abierta en el par → No ejecutar — máx 1 por par.
LONG/SHORT + convicción ≥ 7 + par libre → Ejecutar orden de mercado en Binance
Calcula quantity = $10 / precio. Guarda trade en SQLite. Notifica Telegram.
7
Cada 15 min
Dashboard & Telegram
Escribe dashboard_state.json con todo el estado → sube el JSON a GitHub Gist (si está configurado) → el dashboard PWA lo lee cada 60s.

Telegram solo recibe resumen cada 3 ciclos de análisis (~3h/12h) o cuando hay un trade cerrado. No spamea.
8
Siempre disponible
Cierre manual desde dashboard
El servidor HTTP acepta POST /api/close con trade_id + token. Ejecuta cierre de mercado, notifica Telegram, resetea timer del par.
5 · HMM — Clasificador de régimen de mercado
GaussianHMM · 4 estados · Entrenado por par · Viterbi para predicción
📈 BULL_TREND
Retorno medio positivo
BTC histórico: ~23% del tiempo
Baja volatilidad relativa
→ Claude busca LONGs
📉 BEAR_TREND
Retorno medio negativo
BTC histórico: ~17% del tiempo
Alta volatilidad
→ Claude puede buscar SHORTs
➡️ SIDEWAYS
Retorno ~0
BTC histórico: ~33% del tiempo
Baja volatilidad
→ Claude tiende a NEUTRAL
🔄 REVERSAL
Retorno positivo moderado
BTC histórico: ~26% del tiempo
Alta volatilidad
→ Claude puede entrar con cautela
5 features de entrada al modelo:
log_return retorno logarítmico
vol_rolling_20 volatilidad 20 velas
vol_ratio volumen vs media 20v
RSI_centered RSI normalizado −1/+1
EMA50_slope pendiente EMA50
El régimen se inyecta en el prompt de Claude como contexto — no como regla rígida.
Claude razona sobre él y puede ignorarlo si otros indicadores contradicen.
6 · Escenarios esperables por par (cada ciclo de análisis)
📈 BULL_TREND
Contexto: EMA20 > EMA50, RSI entre 40–65, HMM lleva muchas horas en bull, persistencia alta.

Claude esperado: LONG con convicción 7–9. Target +8%, stop −4%.

Tesis típica: "Tendencia alcista confirmada por HMM (X horas, Y% persistencia). RSI no sobrecomprado. EMA alineadas."

Salidas posibles:
✅ Target (+8%) → WIN
❌ Stop (−4%) → LOSS
🔄 Régimen cambia a BEAR → cierre automático (puede ser WIN o LOSS)
📉 BEAR_TREND
Contexto: EMA20 < EMA50, RSI entre 30–50, HMM en bear sostenido.

Claude esperado: SHORT o NEUTRAL. Si F&G < 20 (miedo extremo), puede preferir no entrar SHORT.

Tesis típica: "Régimen bajista. Tendencia bajista confirmada. RSI no sobrevendido aún."

Salidas posibles:
✅ Target SHORT (precio baja 8%) → WIN
❌ Stop SHORT (precio sube 4%) → LOSS
🔄 Régimen cambia a BULL → cierre automático de SHORT
➡️ SIDEWAYS
Contexto: EMAs cruzadas o muy juntas, RSI alrededor de 50, sin dirección clara.

Claude esperado: Principalmente NEUTRAL. Este régimen no favorece ninguna dirección.

Resultado más probable: 0 tokens gastados en ejecución. El par se "descarta" y se vuelve a analizar en el próximo ciclo de 1h/4h.

Excepción: Si RSI muy bajo (<30) + F&G < 20, Claude puede ver oportunidad contraria.
🔄 REVERSAL
Contexto: Alta volatilidad, cambio reciente de régimen (ej: venía de BEAR). RSI puede estar en extremos.

Claude esperado: Variable. Puede ver LONG si detecta rebote, con menor convicción.

Primera señal real: Claude citó "REVERSAL consolidado (23 barras, 93.6% persistencia)" como tesis de entrada y condición de invalidación. Convicción 8/10.

Salidas posibles:
✅ Si la reversión se confirma → WIN
❌ Si continúa bajando → LOSS por stop
🔄 Cambia a BEAR_TREND → cierre automático

9 · Checklist antes de pasar a producción real
① Observar 1–2 semanas en Testnet. Verificar que stops y targets se detectan correctamente, que el HMM clasifica regímenes razonables, que Claude toma decisiones coherentes.
② Cambiar BINANCE_TESTNET=false en Railway. Solo esto habilita dinero real.
③ Subir ANALYSIS_INTERVAL_MINUTES=240 para reducir costo de tokens en producción.
④ Verificar fondeo: la cuenta de Binance necesita saldo suficiente para los 3 pares simultáneos ($10 × 3 = $30 mínimo más comisiones).
⑤ Setear AGENT_API_TOKEN en Railway para proteger el endpoint de cierre manual.
⑥ Re-entrenar los modelos HMM con datos más recientes si pasaron más de 6 meses desde el entrenamiento original.
